Quantum Rosetta - Design Guidelines
Version: 7.0 (As of 2025-11-13)

This document outlines the current architecture of the Quantum Rosetta project and provides strict guidelines for all future development to ensure the project remains modular, maintainable, and scientifically valid.

==================================
1. Current System Architecture
==================================

The project is a microservice architecture orchestrated by docker compose. It consists of one central "Brain" (rosetta-api) and eight isolated "Hands" (the simulator runners).

1.1. Services (9 Total)
- rosetta-api: The central orchestration, analysis, and UI service.
- pytket-qiskit-runner: Runner for Qiskit Aer (uses Pytket).
- pytket-cirq-runner: Runner for Cirq (uses Pytket).
- pytket-qulacs-runner: Runner for Qulacs (uses Pytket).
- pytket-braket-runner: Runner for the local AWS Braket simulator (uses Pytket).
- pytket-projectq-runner: Runner for ProjectQ (uses Pytket).
- pytket-quest-runner: Runner for QuEST (uses Pytket).
- pennylane-lightning-runner: Runner for Pennylane Lightning (uses native PennyLane QASM parsing).
- pennylane-default-runner: Runner for Pennylane's default.qubit (uses native PennyLane QASM parsing).

1.2. rosetta-api (The "Brain")
The API service is responsible for all orchestration, circuit modification, result analysis, and front-end hosting.

- Smart Configuration: The RUNNER_SERVICES dictionary in main.py is the single source of truth. It defines capabilities ("statevector", "measured_native").
- Dynamic Routing: The API dynamically builds URL lists from the RUNNER_SERVICES config.
- Async Dispatch: Uses an internal _dispatch_to_runners helper to send jobs in parallel.
- Standardized Request Contract:
    - measured_native: Receives a modified_qasm string with explicit measure gates.
    - measured_sampled: Receives original QASM; runner manually samples the statevector.

- Analysis (comparator.py):
    - Correctness (Statevector): Fidelity and Relative Phase.
    - Correctness (Counts): Jensen-Shannon (JS) Divergence.
    - Performance: Execution time (ranking).
    - Resources: Memory usage (ranking by Delta).

- Web UI (MVP): A modular, single-file front-end served from /static/index.html. Displays detailed "Master-Detail" reports.

1.3. Runners (The "Hands")
The runners are isolated services. Their only job is to execute a circuit, perform validated benchmarking, and return raw data.

- Standardized Interface (The 3D Contract): Every runner must implement:
    - POST /run: Returns statevector, execution_time_sec, memory_usage_mb, process_peak_mb, theoretical_memory_mb.
    - POST /run_measured: Returns counts, execution_time_sec, memory_usage_mb, process_peak_mb, theoretical_memory_mb.

- Performance & Resource Benchmarking (Crucial):
    - Time (execution_time_sec): Measured using time.perf_counter() around the core simulation.
    - Memory (Dual Reporting Strategy):
        1. Measured Delta (memory_usage_mb): Uses `qrosetta_commons.helpers.MemoryMonitor` (psutil RSS) to capture the *added* cost. Can be 0 for small circuits.
        2. Process Peak (process_peak_mb): The absolute max RSS of the container.
        3. Theoretical (theoretical_memory_mb): Calculated as `(2^N * 16) / 1024^2`.

1.4. Code Modularity (qrosetta_commons)
Common code is in the shared/ directory, installed as a package.
- qrosetta_commons.models: Pydantic payloads.
- qrosetta_commons.helpers: MemoryMonitor class, calculate_theoretical_memory_mb, _sample_from_statevector.

1.5. Testing
- Tests are located in the `tests/` directory at the project root.
- `test_memory_monitor.py` verifies that the resource profiling tools correctly capture C++ memory spikes.

==================================
2. Future Development Guidelines
==================================

All future additions and modifications must adhere to these rules.

2.1. MUST DOs (The Contract)
- DO follow the Naming Convention ([interface]-[backend]-runner).
- DO register new runners in RUNNER_SERVICES (rosetta-api/main.py).
- DO use `MemoryMonitor` from qrosetta_commons for all memory profiling.
- DO report all three memory metrics: Delta, Peak, and Theoretical.
- DO keep runner-specific logic inside the runner; keep the API generic.
- DO use file-specific volumes in docker-compose for live reloading.

2.2. MUST DON'Ts (The Guardrails)
- DON'T use tracemalloc for simulator profiling (it misses C++ allocations).
- DON'T put test scripts inside the `qrosetta_commons` package; use `tests/`.
- DON'T duplicate the memory calculation formula; import it from helpers.
